# honker Adversarial Review — Findings (2026-05-22)

Scope: `honker-core` (Rust), `honker-extension` (Rust/FFI), and the SQL
contracts in `honker-core/src/honker_ops.rs` that every language binding
inherits. The Python binding was read where it consumes those contracts
(`packages/honker/python/honker/_worker.py`, `_scheduler.py`,
`_honker.py`). Run against the current checkout of branch
`claude/add-honker-adversarial-review-SJurM`. Baseline: `cargo test -p
honker-core` is green (45 passed, 1 ignored).

Process: `.intent/review/adversarial-review-playbook.md`.

Two of the top findings were reproduced with throwaway tests against the
public SQL surface (`honker_enqueue` / `honker_claim_batch` /
`honker_cron_next_after`); the observed output is quoted inline. The
proof tests were not committed.

---

## Summary by risk boundary

- **Queue lifecycle (Track A): one Critical.** The crash path the
  visibility timeout exists for never dead-letters. A job whose worker is
  hard-killed is reclaimed forever, `attempts` growing without bound,
  contradicting the README crash-recovery contract.
- **Cross-process atomicity (Track D): one High, two Medium.** Rate limit
  and lock both do check-then-act across autocommit statements and assume
  a transaction they don't hold. Two processes can both win.
- **Scheduler & cron (Track B): three Medium.** Standard-cron dom/dow
  semantics are wrong; an outage triggers an unbounded catch-up enqueue
  storm; the leader heartbeat refreshes any owner's lock.
- **Wake path & lifecycle (Track C/E): one Medium, two Low.** A
  subscriber that arrives after the watcher thread dies hangs forever; the
  extension's SQL watcher leaks/duplicates threads.
- **Files & FFI (Track F): one Low (latent).** The dead-man's-switch
  `panic!` is only safe under `panic=unwind`.

Net: the engine is careful on the happy path and the SQL is consistently
parameterised, but several invariants are *assumed* rather than
*enforced* at exactly the boundaries the playbook warns about — the
hard-kill path, the second process, and the README's own promises.

1. **F1 — Dead-letter poison jobs on reclaim** (Critical). A crash-looping

## Full findings

### F1 — Crash-looping jobs are reclaimed forever, never dead-lettered

- **Severity:** Critical
- **Confidence:** High (reproduced)
- **Where:** `honker-core/src/honker_ops.rs:461` (`claim_batch`), reclaim
  predicate at `:476-483`. Dead-letter exists only in `retry` at
  `:671-692`. No reaper exists (grep across `honker-core`, `honker-extension`,
  and `packages/honker` confirms the only `attempts >= max_attempts`
  branch is in `retry`).
- **Violated contract:** README "Crash recovery": *"If a worker crashes
  mid-job, the claim expires after `visibility_timeout_s` and another
  worker reclaims. `attempts` increments. After `max_attempts` (default
  3), the row moves to `_honker_dead`."*
- **Concrete bug:** `claim_batch` reclaims expired-`processing` rows with
  the predicate
  `(state='processing' AND claim_expires_at < unixepoch())` and bumps
  `attempts = attempts + 1`, but has **no `attempts < max_attempts`
  guard and no dead-letter step**. The only code that moves a row to
  `_honker_dead` by attempt count is `retry()`, which runs only when a
  *live* worker catches the failure in-process (`_worker.py:104-108`
  `_retry_or_fail`). A worker that is hard-killed (SIGKILL, OOM, panic,
  segfault, power loss) never calls `retry`/`fail`, so the row is
  reclaimed indefinitely.
- **Why it happens in real use:** OOM-killed workers and poison payloads
  that crash the interpreter/runtime are the textbook reason a visibility
  timeout exists. Every reclaim re-hands the poison job to a fresh worker
  that dies again — a self-perpetuating crash loop that also burns a
  claim slot forever and never surfaces in `_honker_dead` for an operator
  to see.
- **Reproduction (observed):** enqueue one job with `max_attempts=3`;
  repeatedly call `honker_claim_batch('q','w',10,-1)` (negative timeout →
  immediately re-expired) without ever acking. After 10 rounds:
  `attempts after 10 crash-reclaims = 10, dead rows = 0`. The documented
  cap (3) is never enforced.
- **Fix (idiomatic):** enforce `max_attempts` on the reclaim path. Either
  (a) in `claim_batch`, before the reclaiming UPDATE, sweep
  over-limit processing rows to dead:

  ```sql
  -- runs in the same call, before the claim UPDATE
  INSERT INTO _honker_dead (id, queue, payload, priority, run_at,
                            max_attempts, attempts, last_error, created_at)
  SELECT id, queue, payload, priority, run_at, max_attempts, attempts,
         'reclaim exhausted (worker crash loop)', created_at
    FROM _honker_live
   WHERE queue = ?2 AND state='processing'
     AND claim_expires_at < unixepoch()
     AND attempts >= max_attempts;
  DELETE FROM _honker_live
   WHERE queue = ?2 AND state='processing'
     AND claim_expires_at < unixepoch()
     AND attempts >= max_attempts;
  ```

  …and add `AND attempts < max_attempts` to the `state='processing'` arm
  of the claim predicate so exhausted rows are not re-handed out; or
  (b) ship a `honker_reap_expired(queue)` function workers call on a
  cadence. Option (a) keeps the guarantee on the hot path with no new
  caller obligation.
- **LLM-style pattern:** Yes — classic. The happy-path failure handler
  (`retry`) was given the dead-letter logic; the crash path that the
  feature exists for was left to "just reclaim", and the README was
  written to describe the intended behaviour rather than the implemented
  one.

### F2 — `rate_limit_try` overshoots under concurrency (TOCTOU)

- **Severity:** High
- **Confidence:** High
- **Where:** `honker-core/src/honker_ops.rs:979-1011`.
- **Violated invariant:** "at most `limit` grants per `per`-second window."
- **Concrete bug:** the function reads
  `SELECT COALESCE(MAX(count),0) … WHERE name=? AND window_start=?`,
  branches `if current >= limit { return 0 }`, then in a *separate*
  statement does `INSERT … ON CONFLICT DO UPDATE SET count = count + 1`.
  Called as `SELECT honker_rate_limit_try(...)` in autocommit, the read
  takes and releases a shared lock before the write transaction begins.
  SQLite serializes write *transactions*, but it does not serialize a
  read in one statement with a write in the next. N processes can all
  read `count = limit-1`, all pass the check, and all increment →
  `count = limit-1+N`. The UPDATE arm has no upper bound, so the overshoot
  is unbounded in the number of concurrent racers.
- **Why it happens in real use:** rate limiting is specifically invoked
  from many workers/requests at once — the contended case is the only
  case that matters, and it's the one that's wrong.
- **Reproduction idea:** two threads/processes, each opening its own
  connection, both call `honker_rate_limit_try('api', 1, 60)` after a
  shared barrier; assert exactly one returns 1. The current code can
  return 1 twice. (Non-deterministic; a loom-style or high-iteration
  stress test makes it reliable.)
- **Fix:** make grant atomic in a single statement and read the effect:

  ```sql
  INSERT INTO _honker_rate_limits (name, window_start, count)
  VALUES (?1, ?2, 1)
  ON CONFLICT(name, window_start) DO UPDATE SET count = count + 1
    WHERE count < ?3
  RETURNING count;
  ```

  Return `1` if a row came back / `changes() > 0`, else `0`. The initial
  insert is fine because `limit > 0` is already validated. The whole
  statement runs under one write transaction, so the check and the
  increment can't be split.
- **LLM-style pattern:** Yes — read-modify-write written as three tidy
  steps that read correctly but aren't atomic.

### F3 — Cron day-of-month / day-of-week uses AND, not standard OR

- **Severity:** Medium
- **Confidence:** High (reproduced)
- **Where:** `honker-core/src/cron.rs:180-191` (`cron_day_matches`) and
  `:88-94` (`CronSchedule::matches`).
- **Violated contract:** `honker_ops.rs:376-378` and the README describe
  the grammar as *"the same 5-field grammar as standard Unix cron."* In
  Vixie cron, when **both** day-of-month and day-of-week are restricted
  (neither is `*`), a tick fires if **either** matches (OR). honker ANDs
  them.
- **Concrete bug:** `sched.days.contains(day) && sched.dows.contains(dow)`
  unconditionally. So `0 0 13 * 5` means "Friday the 13th", not "the 13th
  **or** any Friday."
- **Why it happens in real use:** users porting crontabs from cron/Celery
  beat/systemd timers will silently get far fewer fires than intended for
  any expression that constrains both day fields.
- **Reproduction (observed):** from `2026-01-01`,
  `honker_cron_next_after('0 0 13 * 5', …)` returns
  `next … is 43 days away` (Fri 2026-02-13). Standard cron fires the next
  Friday (within 7 days).
- **Fix:** when both fields are restricted, OR them; when one is `*`, the
  other governs:

  ```rust
  let dom_restricted = self.days.len() != 31;       // or track an explicit flag
  let dow_restricted = self.dows.len() != 7;
  let day_ok = match (dom_restricted, dow_restricted) {
      (true, true)  => self.days.contains(&dom) || self.dows.contains(&dow),
      (true, false) => self.days.contains(&dom),
      (false, true) => self.dows.contains(&dow),
      (false, false) => true,
  };
  ```

  Track "is this field `*`" at parse time rather than inferring from set
  cardinality (a range like `1-31` is not the same as `*` for this rule).
- **LLM-style pattern:** Yes — the single most famous cron subtlety,
  implemented the intuitive (wrong) way, and not covered by any test
  (`cron.rs` tests only exercise dom-or-dow-restricted, never both).

### F4 — Subscribing after the watcher thread dies hangs forever

- **Severity:** Medium
- **Confidence:** Medium
- **Where:** `honker-core/src/lib.rs:985-990` (`subscribe`), `:908-916`
  (`WatcherDeathGuard`), `:1017-1031` (`Drop`).
- **Violated invariant:** "if the watcher dies, every subscriber learns —
  no one blocks forever" (README "Wake backend").
- **Concrete bug:** when the poll thread panics (dead-man's switch on file
  replacement), `WatcherDeathGuard::drop` clears the *current* senders so
  existing receivers disconnect — good. But the `SharedUpdateWatcher` and
  its `senders` map stay alive, and nothing records that the thread is
  dead. A later `subscribe()` inserts a fresh sender into the map; no
  thread will ever fire it, and `recv()` blocks forever with no
  disconnect. (`honker_watcher.join()` would observe the panic, but
  `subscribe()` doesn't consult it.)
- **Why it happens in real use:** a litestream restore / atomic-rename
  deploy trips the dead-man's switch; surviving code that opens a new
  listener (a new SSE connection, a new `db.listen(...)`) on the same
  `Database` object instead of reopening hangs that request thread/task
  indefinitely.
- **Reproduction idea:** extend `shared_update_watcher_signals_subscribers_on_watcher_death`
  — after the panic is observed, call `shared.subscribe()` and assert the
  new `rx.recv()` returns `Err` promptly (currently it blocks).
- **Fix:** add an `AtomicBool alive` set false by `WatcherDeathGuard::drop`
  (and by `close`). `subscribe()` checks it: if dead, return a receiver
  whose sender is dropped immediately (instant `Err`) — or return
  `Result`/`Option` so the binding raises "database watcher closed."
- **LLM-style pattern:** Partly — the death path was carefully handled for
  *existing* subscribers and the *new-subscriber-after-death* case was
  missed.

### F5 — Scheduler heartbeat refreshes any owner's lock (split-brain amplifier)

- **Severity:** Medium
- **Confidence:** Medium
- **Where:** `packages/honker/python/honker/_scheduler.py:325-331`.
- **Violated invariant:** "the leader lease belongs to one owner; only the
  holder may extend it."
- **Concrete bug:** the heartbeat runs
  `UPDATE _honker_locks SET expires_at = unixepoch() + ? WHERE name = ?`
  with **no `AND owner = ?`**. honker's leader election is documented as
  best-effort, but this turns a transient overlap into a sticky one:
  if leader A pauses past the TTL and standby B acquires the lock
  (`lock_acquire` deletes A's expired row and inserts B), A's heartbeat
  then keeps extending **B's** row. A still believes it is leader and
  keeps calling `scheduler_tick`, and A's eventual `lock_release`
  (`WHERE name AND owner=A`, `honker_ops.rs:967-973`) no-ops because the
  row is owned by B. Result: two live schedulers firing duplicate jobs,
  and the lock never returns to a clean single-owner state on its own.
- **Why it happens in real use:** GC pauses, VM migration, a long
  synchronous handler on the scheduler process — exactly the conditions
  TTL leases are meant to survive.
- **Reproduction idea:** insert a `_honker_locks` row owned by "B" with a
  future expiry, run one heartbeat iteration as "A", assert the row's
  `owner` is unchanged and that A detects it no longer holds the lock
  (currently the row's expiry is extended and A is none the wiser).
- **Fix:** `… WHERE name = ? AND owner = ?`, and treat
  `changes() == 0` as "lost leadership" → stop the loop / re-contend.
- **LLM-style pattern:** Yes — the renew was written as "keep the named
  lock fresh" without re-checking ownership.

### F6 — Scheduler catch-up replays every missed boundary in one writer-held call

- **Severity:** Medium
- **Confidence:** High
- **Where:** `honker-core/src/honker_ops.rs:1104-1159` (`scheduler_tick`),
  loop at `:1128`.
- **Violated invariant:** "bounded work per tick" / single-writer
  availability.
- **Concrete bug:** for every due task, `while next_fire_at <= now_unix`
  enqueues one job *per missed boundary* and writes one
  `_honker_notifications` row each. After an outage this is intentional
  catch-up, but it is unbounded: an `@every 1s` (or `* * * * *`) schedule
  that was down for a day enqueues ~86,400 (or 1,440) jobs in a single
  scalar-function call that holds the one writer connection the entire
  time, doubling the row count via wake notifications.
- **Why it happens in real use:** leader failover, a deploy, or a laptop
  resuming from sleep all produce a large gap; the first surviving leader
  pays the entire bill synchronously and the queue floods.
- **Reproduction idea:** register `@every 1s`, set its
  `next_fire_at` a day in the past, call `honker_scheduler_tick(now)`, and
  assert the resulting `_honker_live` count and call duration are bounded
  (they aren't).
- **Fix:** cap catch-up (`MAX_CATCHUP_FIRES`, e.g. 1) and/or fast-forward
  to the next future boundary for interval schedules — fire at most once
  for the missed window and advance `next_fire_at` past `now`. Make the
  policy explicit (cron beat's "coalesce missed runs").
- **LLM-style pattern:** Partly — the catch-up loop is correct for a small
  gap and pathological for a large one; no upper bound was considered.

### F7 — `lock_acquire` by the current owner returns success but does not refresh the TTL

- **Severity:** Medium
- **Confidence:** High
- **Where:** `honker-core/src/honker_ops.rs:937-965`.
- **Violated invariant:** "re-acquiring a lock you hold renews your lease."
- **Concrete bug:** for a held, non-expired lock, `DELETE … WHERE
  expires_at <= unixepoch()` removes nothing, `INSERT OR IGNORE` is
  ignored (the `name` PK exists), and the final
  `SELECT owner` matches → returns `1`. But `expires_at` is left at the
  original value. A caller that calls `acquire(name, me, ttl)` periodically
  to renew its lease sees success while the lease silently counts down and
  expires, letting another owner steal it.
- **Why it happens in real use:** "call acquire again to extend" is the
  obvious renew pattern, and `1` (success) actively confirms the wrong
  mental model. honker's own scheduler avoids this only by using a raw
  `UPDATE` instead (see F5) — so the foot-gun is live for every *other*
  lock user.
- **Reproduction idea:** `lock_acquire('x','me',100)`, sleep,
  `lock_acquire('x','me',100)`, assert `expires_at` advanced (it doesn't).
- **Fix:** make acquire an upsert that renews on same-owner-or-expired:

  ```sql
  INSERT INTO _honker_locks (name, owner, expires_at)
  VALUES (?1, ?2, unixepoch() + ?3)
  ON CONFLICT(name) DO UPDATE SET owner = excluded.owner,
                                  expires_at = excluded.expires_at
    WHERE _honker_locks.owner = excluded.owner
       OR _honker_locks.expires_at <= unixepoch();
  ```

  then `SELECT owner` and compare as before. This also collapses the
  current delete-then-insert-then-select into one atomic statement.
- **LLM-style pattern:** Yes — "return 1 if we own it" conflated holding
  with renewing.

### F8 — `N/step` in a cron field yields only `N`

- **Severity:** Low
- **Confidence:** Medium
- **Where:** `honker-core/src/cron.rs:131-176` (`parse_field`).
- **Concrete bug:** a part like `5/15` splits into `range_part="5"`,
  `step=15`; since `range_part` is neither `*` nor a range, it becomes
  `(start,end)=(5,5)`, so the step never iterates and the field is just
  `{5}`. Standard cron treats `5/15` as `5-max/15` (i.e. `5,20,35,50`).
- **Why it happens in real use:** `*/15` works (because `*` expands to
  `lo..hi`), so users reasonably expect `30/15` to mean "from 30 by 15";
  it silently means "exactly 30."
- **Reproduction idea:** `next_after_unix("30/15 * * * *", t)` should be
  able to land on minute 45; it never will.
- **Fix:** when a bare integer precedes `/step`, set `end = hi`.
- **LLM-style pattern:** Yes — `*` and ranges were handled; the bare-number
  + step combination was overlooked.

### F9 — Extension SQL watcher spawns a poll thread per `open`

- **Severity:** Low
- **Confidence:** High
- **Where:** `honker-extension/src/lib.rs:69-81` (`open_watcher_handle`),
  called from `honker_update_watcher_open` (`:84-98`).
- **Violated design goal:** README "Wake path": *"100 subscribers = 1
  poll thread."*
- **Concrete bug:** each `honker_update_watcher_open` builds a brand-new
  `SharedUpdateWatcher::new_with_config`, which spawns its own poll thread
  and read-only connection. For the pure-SQL / ORM integration path, N
  listeners = N threads each running `PRAGMA data_version` at 1 kHz on the
  same file — the exact stat/poll storm `SharedUpdateWatcher` was built to
  collapse (the in-process bindings share one because they hold one
  `SharedUpdateWatcher` per `Database`).
- **Why it happens in real use:** an app that opens a watcher per SSE
  connection through the loadable extension regresses to per-listener
  threads.
- **Fix:** key a `HashMap<(PathBuf, backend), Arc<SharedUpdateWatcher>>`
  in the static and `subscribe()` per open; close the shared watcher when
  its last subscriber goes away (refcount).
- **LLM-style pattern:** Partly — the sharing primitive exists and the
  extension path just doesn't use it.

### F10 — `honker_update_watcher_wait` leaks the watcher if `close` races the wait

- **Severity:** Low
- **Confidence:** Medium
- **Where:** `honker-extension/src/lib.rs:99-136`.
- **Concrete bug:** `wait` does `SQL_WATCHERS.lock().remove(id)`, releases
  the lock, blocks on `recv_timeout`, then re-inserts the handle. While it
  is blocked the id is absent from the map, so a concurrent
  `honker_update_watcher_close(id)` finds nothing and no-ops; when `wait`
  returns it re-inserts the handle, and the underlying poll thread +
  connection live forever (no later `close` will find a "closing" marker).
- **Why it happens in real use:** any code path where one statement waits
  and another closes the same watcher id (shutdown racing a poll).
- **Fix:** don't remove during the wait — hold the handle in the map and
  block on a cloned `Receiver`, or set a `closing` flag the post-wait
  re-insert checks (and then closes instead of re-inserting).
- **LLM-style pattern:** Yes — "take it out so we don't hold the global
  lock while blocked" introduced a lifecycle hole.

### F11 — Dead-man's-switch `panic!` is only safe under `panic=unwind`

- **Severity:** Low (latent; severity High if any binding builds `panic=abort`)
- **Confidence:** Medium
- **Where:** `honker-core/src/lib.rs:776`, `shm_watcher.rs:215`,
  `kernel_watcher.rs:168`.
- **Violated contract:** README: file replacement → *"they shut down
  loudly — every subscriber sees an error from `update_events()` instead
  of hanging."* That contract depends on stack unwinding so the closure
  (and thus `WatcherDeathGuard`) drops and subscribers disconnect.
- **Concrete bug:** the detection uses `panic!`. Under
  `panic=unwind` it is contained to the watcher thread (correct, tested).
  Under `panic=abort` — a profile bindings can set for size/speed — a
  routine, expected event (litestream restore, atomic-rename deploy)
  aborts the entire host process (the Python interpreter, the Node
  runtime). The "loud failure" becomes "kill the app."
- **Why it happens in real use:** file replacement is an explicitly
  supported scenario; the failure mode is environmental (the binding's
  release profile), so it won't show up in `honker-core`'s own tests.
- **Fix:** don't panic for an expected condition. Set a dead flag, clear
  senders (so subscribers disconnect on every panic strategy), log, and
  `return` from the loop. Reserve `panic!` for genuine invariant
  violations. This also composes with F4's `alive` flag.
- **LLM-style pattern:** Partly — "fail loudly" was implemented as a panic
  without considering the abort profile or FFI host.

### F12 — `scheduler_tick` hardcodes `max_attempts = 3` for scheduled jobs

- **Severity:** Low
- **Confidence:** High
- **Where:** `honker-core/src/honker_ops.rs:1131-1134` (the `enqueue`
  call passes a literal `3 /* max_attempts default */`).
- **Concrete bug:** scheduled fires always enqueue with `max_attempts=3`
  regardless of the queue's or schedule's configured retry policy. A queue
  built with a different `max_attempts` silently gets 3 for cron-fired
  jobs.
- **Fix:** store `max_attempts` on `_honker_scheduler_tasks` (default 3)
  and forward it; or document the fixed value explicitly.
- **LLM-style pattern:** Yes — a default inlined at the call site instead
  of threaded through.

## Verified non-bugs (recorded so they aren't re-investigated)

- **Stream offsets cannot skip under concurrent multi-connection publish.**
  `_honker_stream.offset` is `AUTOINCREMENT`; offsets are assigned at
  INSERT. SQLite permits only one write transaction at a time across the
  whole db, so a later writer cannot allocate and commit a higher offset
  before an earlier writer commits a lower one — no
  "consumer advances past an uncommitted lower offset" gap. The
  at-least-once stream contract holds. (`honker_ops.rs:1388-1408`,
  `1413-1458`.)
- **No id reuse breaks listener cursors.** `_honker_notifications.id`,
  `_honker_live.id`, and `_honker_stream.offset` are all
  `INTEGER PRIMARY KEY AUTOINCREMENT`, so deletes/prunes never recycle a
  value below a listener's `id > last_seen` / `offset > saved` cursor.
- **No SQL injection.** Every query uses bound parameters; channel names
  are built with `format!("honker:{queue}")` but passed as a *bound* value
  (`honker_ops.rs:590-612`, `705-710`, `1401-1406`), never interpolated.
- **`json_str` produces valid JSON.** It quotes, escapes `"` `\` `\n` `\r`
  `\t` and `< 0x20` controls, and emits the rest of the (always-valid
  UTF-8) Rust string raw. No injection via payload/key
  (`honker_ops.rs:1500-1518`).
- **Bounded subscriber channel (cap=1) does not lose wakes.** A commit
  after a consumer's SELECT either delivers a tick or coalesces into a
  still-pending tick the consumer will drain on its next `recv()`; the
  1 ms poll and the binding's 5 s paranoia poll are backstops
  (`lib.rs:953-963`, README "Claim iterator").
- **`shm_watcher` byte order is correct.** The WAL index is stored in
  native byte order, and the watcher uses `from_ne_bytes` and refuses
  big-endian (`shm_watcher.rs:65-68, 95, 174-180`).
- **`scheduler_tick` cannot infinite-loop.** `next_after_unix` advances by
  ≥1 s for cron (`cron.rs:211`) and by `interval_s > 0` for `@every`
  (`:111-113, 331-333`), so the catch-up `while` always terminates (it is
  merely unbounded in volume — see F6).

  dead-letters"** — F1 (crash path never dead-letters).

- **"A lease belongs to one owner; only the holder extends it"** — F5, F7.
- **"Standard Unix cron semantics"** — F3, F8.
- **"Bounded work per tick / the single writer is not monopolised"** — F6.
- **"If the watcher dies, every subscriber learns; no one hangs"** — F4;

## Areas needing deeper review

- **Wake loss during watcher reconnect (`lib.rs:744-764`).** On a
  non-transient poll error the loop drops the connection, fires one
  conservative `on_change`, then re-baselines `last_version` on reconnect.
  Commits during the reconnect-failing window rely solely on the 5 s
  paranoia poll. Worth a fault-injection test (revoke fd / chmod) to
  confirm no permanent silent gap.
- **`heartbeat` / `cancel` vs in-flight `retry` interleavings.** The
  single-statement guards look correct, but a property test over
  {claim, ack, retry, fail, cancel, heartbeat, reclaim} interleavings
  would prove exactly-once settlement (and would have caught F1).
- **DST fall-back ambiguity.** `next_after_unix_in_tz` always picks the
  earlier of an ambiguous local time (`cron.rs:336-347`); confirm against
  the intended semantics for a job scheduled inside the repeated hour
  (currently it can fire once where some crons fire twice).
- **Cross-binding schema/feature drift.** This review read the Rust core
  and the Python consumer; the Node/Go/Ruby/JVM/etc. bindings should be
  checked for the same crash-path assumptions (F1) and renew patterns
  (F7).
- **`_honker_notifications` unbounded growth.** Every enqueue / retry /
  stream publish writes a wake row that is never auto-pruned (by design),
  but combined with F6 the growth is super-linear after an outage.
  Consider a default bounded-retention helper.
